## Р-11. Раздел «Релизы» в меню бота

- **Статус:** 🔶 ФОРМАТ СОГЛАСОВЫВАЕТСЯ (2026-07-16). Владелец решил:
  разделу быть (не гипотеза). Формат согласован владельцем + Claude +
  ChatGPT, ниже. Код — по «да» владельца после трёх последних вопросов.
- **Зачем:** новый контент в боте (медиа), музыканты показывают работы
  (сообщество), автор сам делится карточкой — реклама бота; честная
  витрина студий через поле «где записано»; рубрика для дайджеста.

**Согласованный формат (MVP):**
- Меню: 🎵 Релизы → Новые релизы / Добавить релиз / Мои релизы.
- Карточка: обложка (обязательно), исполнитель, название, тип
  (сингл/EP/альбом/клип/live), дата, жанр (необязательно), описание
  (необязательно), «где записано» (необязательно), ссылки на площадки —
  несколько подписанных, хотя бы одна обязательна (YouTube, Spotify,
  Яндекс, Apple, Bandcamp, SoundCloud, VK, другая).
- Лента по свежести, без городов и без рубрик по жанрам (пока база мала).
- Архитектура: общая таблица listing (type='release') + своя мета-таблица
  по образцу Афиши (events_meta). Без 30-дневного срока, продлений,
  карантина и «продано» — свои статусы: опубликован / скрыт / удалён
  (+ «на модерации/отклонён», если владелец выберет премодерацию).
- Публикуем только официальное: синглы, EP, альбомы, клипы, live-релизы.
  Демки, «оцените сведение» — не сюда (позже отдельная рубрика
  «Творчество»). Галочка автора: «я автор/представитель или имею
  разрешение». Файлы не храним — только обложка и внешние ссылки.
- Бесплатно; без рейтингов, голосований, комментариев, чартов, плеера.
  Монетизация вокруг релизов — только позже, по общим правилам.
- Релизы не считаются в лимит 5 бесплатных объявлений.

**Решено владельцем 2026-07-16:** (1) пункт меню — «🎵 Релизы»;
(2) модерация: релиз виден сразу, админу уведомление с кнопкой «Скрыть»;
гибрид «первый релиз автора — с проверкой, дальше сразу» — записанное
усиление на случай мусора; (3) кнопки связи с автором публикации нет —
позже путь «релиз → Об исполнителе → карточка → Связаться, если артист
разрешил». Плюс кнопка «Пожаловаться» на карточке релиза с первого дня.

**Медиа в релизе (решения владельца 2026-07-16):**
- Можно приложить трек, клип или **альбом — список аудиофайлов**:
  пользователь шлёт треки по одному, бот хранит список file_id в порядке
  загрузки, при просмотре отдаёт пачками по 10 (media group). Файлы
  загружает пользователь (лимит Telegram 2 ГБ); пересылка по file_id без
  ограничений размера; наше хранилище не используется. Ограничен только
  предпросмотр в веб-админке (getFile — 20 МБ).
- **YouTube играет внутри бота**, как в барахолке/услугах: ссылка идёт в
  текст карточки, Telegram разворачивает встроенный плеер — без перехода
  в приложение YouTube. Плеером становится одна ссылка на сообщение
  (клип приоритетнее), остальные площадки — кнопками.
- Правило: хотя бы одна ссылка ИЛИ файл.
- Проверено на живом клиенте: окно «Перейти по ссылке?» Telegram
  показывает для ссылок с ПОДМЕНЁННЫМ текстом и для URL-кнопок. ГОЛЫЙ
  адрес в тексте (как в барахолке: «Видео: https://…») открывается сразу,
  без подтверждений — поэтому YouTube в карточке релиза живёт голой
  ссылкой в тексте, а не кнопкой.

**Финальные технические решения (2026-07-16, согласованы владельцем +
Claude + ChatGPT; стройка одобрена):**
- Альбом — отдельная таблица треков (release_track: позиция, название,
  file_id, file_unique_id, длительность), НЕ список file_id в одном поле.
- Хранить file_id И file_unique_id (file_id привязан к конкретному боту;
  задел на сеть ботов).
- Показ альбома: карточка с трек-листом, треки отправляются по выбору;
  «отправить весь альбом» — позже (медиагруппы по 10).
- YouTube: тот же механизм, что в барахолке (ссылка в тексте карточки →
  встроенный плеер Telegram) + резервная кнопка «Открыть на YouTube»
  (ролики с ограничениями во встроенном плеере не играют).
  YouTube-плейлист допустим как ссылка на альбом/EP.
- Связь событие↔исполнитель — позже, многие-ко-многим (event_artist),
  предложение «связать с вашим исполнителем?» только управляющему.
  Без автосвязи по совпадению названий.
- Публикация сразу для всех (бот ещё не запущен публично); выключатель
  releases_enabled — как аварийный рубильник, включён.

**Поправка архитектуры (по разбору ChatGPT, принята):** исполнитель —
отдельная сущность С ПЕРВОЙ арки, а не «текстом, потом соберём».
Причины: один человек ведёт несколько проектов; одну группу публикуют
разные люди; релиз может выкладывать менеджер/лейбл. Автосборка по
owner_id дала бы кашу. В арке релизов создаётся минимальный Artist
(название, тип: сольный/группа/дуэт/проект/DJ/другое, фото, управляющий
пользователь); в мастере релиза шаг «выбрать моего исполнителя / создать
нового»; релиз хранит artist_id сразу.

## Р-14. Операционная модель при росте (география, сеть ботов, делегирование ИИ)

- **Статус:** 💡 ПРИНЯТО КАК НАПРАВЛЕНИЕ (2026-07-19, обсуждение владелец +
  Claude). Код и инфраструктура — по «да» владельца, по фактам, не по календарю.
- **Задача:** при расширении (другие страны, несколько ботов, новые функции)
  один человек утонет, если делать руками то, что растёт с объёмом. Нужна
  структура: работу-объём — автоматизировать/делегировать, за владельцем
  оставить только суждение.

**Как масштабироваться — размножением, не расшариванием:**
- Каждому региону — свой самодостаточный экземпляр: свой бот, свой SQLite,
  своя машина/контейнер; управление централизованное. Общей базы нет.
- Владелец подтвердил 2026-07-19: **Казахстан — отдельный бот** (не общий с
  Сербией). Это и есть первая копия шаблона.
- **Важно про формулировку.** Фраза «всё на одной машине, один SQLite, один
  процесс бота» — сжатие Claude двух строк CLAUDE.md, а не дословное правило
  владельца. Разделять:
  - «Один SQLite, не делить по сети» — **технический факт** выбора SQLite
    (встроенная однофайловая база), а не прихоть. С моделью «своя база на
    регион» он не конфликтует — наоборот, подсказывает её.
  - «Один процесс / одна машина» — **следствие текущей реализации** (замки
    FSM внутрипроцессные) + выбор простоты; пересматриваемо при переархитектуре.
- Практический вывод: Казахстан (бот №2) — момент задать шаблон
  мульти-региона **осознанно**: код параметризуется по региону (конфиг),
  а не форкается. Что заложим для №2 — образец для №N.

**Карта «оставить / делегировать»:**
- Владельцу (суждение): партнёрства, спорная модерация, стратегия, деньги, право.
- Автоматизировать/делегировать (объём): поток модерации, дайджесты, рассылки,
  отчёты, мониторинг живости ботов.

**Граница делегирования ИИ (безопасность всей схемы):**
- **ИИ сам** — обратимое, низкая ставка: черновики (дайджесты, ответы,
  реклама), пересказ аналитики в человеческие отчёты, первичная сортировка
  модерации (пометить подозрительное), мониторинг/алерты.
- **ИИ предлагает — владелец решает** — необратимое, спорное, юр./деньги,
  конфликты между людьми, пограничная модерация.

**Руль — измерение:** уже есть `first_source` и поток аналитики. При росте —
пор-региональные сводки с ИИ-пересказом «что важно за неделю»: один человек
управляет многими ботами, читая выводы, а не сырые данные.

**Про инструменты автоматизации (n8n и подобные):**
- Внутренние задачи над своей базой (модерация, дайджесты, напоминания) — в
  процессе бота: у него уже есть планировщик (lifecycle_worker), доступ к
  SQLite и объекту Bot; внешний движок тут дерётся с «один процесс / не делить
  SQLite» и дублирует то, что и так есть.
- Фильтр мата/оскорблений — **встроенная функция** в точках проверки текста
  (как проверки «не пустое»/«длина»): нормализация (гомоглифы, разделители) +
  курируемый список; сначала **пометка для ревью в админке**, жёсткий блок —
  только если мусора реально много. Умный уровень (LLM-классификатор) — позже
  и **прямым вызовом из Python**, не через n8n.
- Внешняя раздача (соцсети, реклама бота) — единственная честная зона внешних
  средств, и по времени на стратегии (этап привлечения). Но: сначала руками,
  измерить по `first_source`, что конвертит, потом автоматизировать победителя;
  один Telegram-канал — в самом боте; несколько капризных площадок — тогда
  внешний инструмент, через API бота, **никогда не в SQLite**. Учесть, что это
  ещё одно хранилище токенов (поверхность безопасности).

- **Когда вернуться:** перед стройкой второго бота (Казахстан) — задать шаблон
  мульти-региона и делегирования; отдельные куски (фильтр, отчёты, автопостинг)
  — по мере появления реальной боли, а не заранее.
- **Связано:** гипотезы «переносимый профиль между странами» и «локальные хабы»
  (поздний резерв, ниже).

## Р-15. Ответ на обращения обратной связи (через бота)

- **Статус:** ✅ РЕШЕНО и реализовано MVP (2026-07-20, владелец + Claude).
- **Проблема:** бот обещал ответ на обращение, но канала ответа не было; по
  @username достучаться нельзя, если его нет.
- **Решение (MVP):**
  - После обращения — тёплое подтверждение (финальный текст владельца) +
    кнопки «🔔 Нужен ответ» / «Ответ не нужен».
  - «Нужен ответ» → `feedback.needs_reply=1`, пинг админам.
  - Ответ администратора — **через бота**: в уведомлении кнопка «✍️ Ответить»,
    админ пишет текст, бот доставляет пользователю в его чат (по `user_id`,
    username не нужен), метит `answered_at`; недоставка (блок бота) —
    честно сообщается админу.
  - Миграция: `feedback.needs_reply`, `feedback.answered_at`, `feedback.answer_text`
    (в `init_db`).
  - «Мои обращения» (раздел «Обратная связь» → выбор «Написать нам» / «Мои
    обращения»): пользователь в любой момент перечитывает свой вопрос и ответ
    администратора — сам ответ в чате по-прежнему авто-удаляется при навигации
    (гигиена чата), но история сохранена отдельно; можно удалить своё
    обращение (с подтверждением).
  - Админ-панель (Telegram): «Неотвеченные» — отдельный пункт сразу под
    списком, сортировка от старых к новым (подстраховка от «пропустил в общем
    списке»); единое уведомление на обращение редактируется на месте (не
    плодится), кнопка «Убрать»; быстрый ответ — админ может просто напечатать
    текст, не нажимая «Ответить», если есть «вооружённое» обращение.
  - Веб-админка (`category_admin.py`, вкладка «✉️ Обратная связь») — тот же
    функционал зеркалом: список/фильтр/пагинация/карточка/ответ через прямой
    вызов Telegram Bot API (веб-админка — отдельный процесс, к боту не
    привязана)/удаление. Общая таблица `feedback` — источник правды один,
    синхронизация между Telegram и веб не нужна.
  - Тесты: `tests/test_feedback_reply.py` (13) + `tests/test_category_admin_feedback.py` (10).
- **Правило голоса:** все пользовательские тексты — на «Вы», никогда на «ты»
  (личное требование владельца).
- **Отложено до фактов (правило 5):** «личка» — открытый двусторонний диалог.
  Технически несложно (тот же канал), но операционно рискованно: превращает
  обратную связь в живую поддержку на одном человеке (риск «утонуть», см.
  [[Р-14]]). Сначала MVP → посмотреть реальный объём «Нужен ответ» → затем,
  если поток спокойный, включить **ограниченную ветку** (follow-up только
  после того, как админ уже ответил), а не открытый чат. Код MVP написан с
  заделом под ветку (есть `id` обращения и `chat_id`).
- **Когда вернуться:** через 2–4 недели живого потока — оценить объём.

## Журнал гипотез

| Гипотеза | Категория | Проверка |
|---|---|---|
| Донаты как репетиция платёжного контура | гипотеза | включить кнопку, смотреть техпрохождение, не доход |
| Антикафе как рекомендуемое место встречи | гипотеза (пилот) | объявить, считать использования |
| Консьерж (ручное сведение) | гипотеза (пилот, лимиты) | несколько случаев/нед, доля успешных |
| Призы свободными часами студии | гипотеза (пилот) | розыгрыш с объективными условиями |
| Добровольные двуязычные объявления (SR) | гипотеза | доля авторов, добавивших перевод |
| Вопрос «Удалось ли?» | принято в стратегию (слой 1) | доля ответов, распределение |
| Дайджест | принято в стратегию (слой 1) | открытия, возвраты после выпуска |
| Спрос-объявления `intent=request` | принято в стратегию (слой 1) | доля «ищу», их контактность |
| Переносимый профиль между странами | поздний резерв | — |
| Локальные хабы в других городах | поздний резерв | — |
